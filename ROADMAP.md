## A. 過濾增強

### A1. 頻道影片數量過濾

**描述**：過濾訂閱者數量低於某個門檻的頻道影片（例如：訂閱數 < 1000 的頻道）。

**技術可行性分析**：
- ⚠️ YouTube DOM 中「訂閱數」**不顯示**於首頁影片卡片，需要額外呼叫 YouTube 內部 API（如 `innertube`）或從影片頁面取得。
- 高實作成本，且 API 可能隨時失效。

**複雜度**：🔴 高  
**影響面**：🟡 中  
**備注**：若不依賴 API，只能以「頻道名稱黑名單」替代。

---

### A2. 標題評分系統（Clickbait 偵測）

**描述**：用啟發式規則替影片標題評分，高度 Clickbait 的影片（例：過多驚嘆號、全大寫、emoji 數量）自動隱藏。

**技術可行性分析**：
- 🟢 可在 `filter-engine.ts` 新增 `getFilterClickbait()` 方法。
- 規則可以是純 JS（計算標點符號數量、大寫率、emoji 數量等）。
- 誤殺率是主要風險，需要細心調教閾值。

**複雜度**：🟡 中  
**影響面**：🟢 大  
**實作建議**：
```typescript
// 可能的實作樣板
function scoreTitle(title: string): number {
    let score = 0;
    score += (title.match(/[!！]/g) || []).length * 2;
    score += (title.match(/[\p{Emoji}]/gu) || []).length;
    score += title === title.toUpperCase() ? 5 : 0;
    return score;
}
```

---

### A3. 根據上傳日期過濾（超舊影片過濾）

**描述**：過濾超過 N 年前上傳的影片（例如：超過 5 年前的影片不顯示）。

**技術可行性分析**：
- 🟢 `LazyVideoData` 已有 `timeAgo` 欄位（分鐘數），可直接比較。
- 目前 `GRACE_PERIOD_HOURS` 是豁免**新**影片；反向亦可豁免**舊**影片。
- 需在 `ConfigManager` 新增 `MAX_AGE_DAYS` 設定。

**複雜度**：🟢 低  
**影響面**：🟡 中  
**備注**：對歷史存檔性 Channel 的觀眾可能不友好，應預設關閉。

---

### A4. 短標題過濾（標題過短的影片）

**描述**：標題少於 N 個字元的影片視為低品質，自動過濾。

**技術可行性分析**：
- 🟢 邏輯極簡單，`title.length < threshold` 即可。
- 需在 `FilterEngine` 新增對應方法，在 `ConfigManager` 新增 `TITLE_MIN_LENGTH` 設定。

**複雜度**：🟢 低  
**影響面**：🟡 中  
**備注**：中文標題天生較短，需與英文分開設閾值或以字元數計算（不適合用空白數計算詞數）。

---

### A5. 多語言頻道名稱匹配改進

**描述**：目前頻道黑名單以字串精確比對（或 OpenCC 輔助繁簡），對日文、韓文頻道名稱的支援不足。考慮支援正規表達式直接輸入。

**技術可行性分析**：
- 🟢 `_compileList()` 已有部分邏輯，可擴充支援使用者輸入 `regex:xxx` 格式直接傳入 RegExp。
- 需修改 `config.ts` 的 `_compileList()` 邏輯。

**複雜度**：🟢 低  
**影響面**：🟡 中  

---

### A6. 播放清單（Playlist）章節過濾

**描述**：使用者在播放清單頁面（`/playlist?list=...`）時，仍可對清單內的影片套用關鍵字/頻道過濾。

**技術可行性分析**：
- 目前 `isPageAllowingContent` 方法讓播放清單頁面的過濾全部停用（避免誤刪）。
- 可以改成「部分過濾」：允許關鍵字黑名單生效，但停用觀看數/時長過濾。
- 需細分 `isPageAllowingContent` 的「強制跳過」與「部分跳過」邏輯。

**複雜度**：🟡 中  
**影響面**：🟡 中  

---

### A7. Shorts 頁面支援（`/shorts/xxxx` 路由）

**描述**：目前腳本主要針對首頁、搜尋頁、頻道頁。進入 Shorts 全螢幕模式後，腳本沒有任何作用（也沒辦法）。

**技術可行性分析**：
- 🔴 Shorts 的全螢幕模式 DOM 結構完全不同，是獨立的 `ytd-shorts` 元件。
- 若要「跳過某頻道的 Shorts」，需要偵測 `ytd-shorts` 的頻道資訊並攔截滑動事件，技術難度極高。
- 可先從「隱藏 Shorts 全螢幕下方的評論/分享按鈕」等小功能入手，降低複雜度。

**複雜度**：🔴 高  
**影響面**：🟡 中

## B. 白名單與訂閱系統

### B1. 訂閱名單匯出功能

**描述**：允許使用者將腳本自動掃描到的訂閱頻道名單匯出成純文字清單，方便確認或手動複製到頻道白名單。

**技術可行性分析**：
- 🟢 `SubscriptionManager.subscribedSet` 已是 `Set<string>`，直接轉陣列後 `join('\n')` 即可。
- 在系統選單新增「匯出訂閱名單」按鈕，呼叫 `GM_setClipboard` 即完成。

**複雜度**：🟢 低  
**影響面**：🟡 中  

---

### B2. 訂閱名單手動管理介面

**描述**：讓使用者可以在設定選單中**手動新增/移除**訂閱頻道保護名單，而不只依賴自動掃描。

**技術可行性分析**：
- 🟢 `SUBSCRIBED_CHANNELS` 已是 `ConfigState` 中的 `string[]`，只是選單尚未暴露其編輯入口。
- 在 `UIManager.showListMenu()` 新增對 `SUBSCRIBED_CHANNELS` 的 `manage()` 呼叫即可。

**複雜度**：🟢 低  
**影響面**：🟡 中  

---

### B3. 白名單規則測試工具

**描述**：讓使用者輸入一個頻道名稱或標題，即時顯示「這個影片會被哪條規則過濾」或「被哪條白名單豁免」。

**技術可行性分析**：
- 🟢 核心邏輯已在 `FilterEngine` 中，只需包裝一個 `simulateFilter(channelName, title, viewCount)` 函式。
- UI 透過 `prompt` + `alert` 呈現結果即可。

**複雜度**：🟡 中  
**影響面**：🟢 大（對新手使用者非常友善）  

---

### B4. 基於「已訂閱」自動建立白名單

**描述**：在使用者確認後，自動將所有已掃描到的訂閱頻道名稱複製進「頻道白名單」，讓訂閱頻道完整享有白名單保護（目前訂閱保護只針對弱規則有效）。

**技術可行性分析**：
- 🟢 邏輯很簡單：`Array.from(subManager.subscribedSet)` → 寫入 `CHANNEL_WHITELIST`。
- 但可能讓白名單非常大，需提示使用者確認。

**複雜度**：🟢 低  
**影響面**：🟡 中

## C. 使用者介面 (UI/UX)

### C1. 自訂 Floating Panel UI（取代 prompt/alert）

**描述**：設計一個可拖曳的浮動設定面板，取代目前基於 `prompt`/`alert` 的原生對話框 UI，提供更好的操作體驗（多欄位同時設定、即時預覽等）。

**技術可行性分析**：
- 🔴 高複雜度。需要注入 HTML/CSS，並確保不與 YouTube 現有樣式衝突（YouTube 使用大量 Shadow DOM）。
- 需要考慮 CSP（Content Security Policy）限制。
- 一旦 YouTube 改版，UI 可能出現視覺錯位。

**複雜度**：🔴 高  
**影響面**：🟢 大  
**備注**：這是一個「大工程」，若要做應該單獨一個主要版本（v3.0）。

---

### C2. 選單加速（記憶上次選單位置）

**描述**：使用者常常重複進入同一個子選單（例如總是進「名單管理 → 頻道黑名單」），應該記憶上次操作的路徑，或提供快捷方式。

**技術可行性分析**：
- 🟡 可在 `GM_setValue` 記憶 `lastContext`，下次開啟選單時直接跳到上次的位置。
- 需修改 `UIManager.showMainMenu()` 的啟動邏輯。

**複雜度**：🟡 中  
**影響面**：🟡 中  

---

### C3. 即時計數（Overlay Badge）

**描述**：在 Tampermonkey 圖示上或頁面固定位置顯示「本次已過濾 N 個」的數字，讓使用者不必進選單就能看到效果。

**技術可行性分析**：
- `GM_setValue` 可以配合 `@badge` Tampermonkey 功能顯示數字，但此功能支援程度因瀏覽器而異。
- 簡單的替代方案：在頁面右下角注入一個小 badge div。

**複雜度**：🟡 中  
**影響面**：🟡 中  

---

### C4. 過濾記錄面板（日誌查看器）

**描述**：目前過濾記錄只輸出到 console（Debug Mode 開啟時）。新增一個設定選單選項，可以查看最近 N 筆被過濾的影片列表（標題、頻道、原因）。

**技術可行性分析**：
- 🟢 需在 `FilterStats` 中改為記錄每筆過濾的詳細資訊（目前只記錄 reason 計數）。
- 需要限制記錄筆數（如最多 100 筆），避免記憶體無限增長。
- UI 透過 `alert` 多行文字呈現即可（不需要複雜 UI）。

**複雜度**：🟡 中  
**影響面**：🟡 中  

---

### C5. 多套設定檔（Profiles）

**描述**：允許使用者儲存多套設定（例如「工作模式」、「休閒模式」），快速切換不同過濾強度。

**技術可行性分析**：
- 🟡 需要在 GM Storage 中以 profile name 為 key 儲存多份 ConfigState JSON。
- 選單需新增「切換 Profile」入口。
- 需要處理 profile 之間的合併/覆蓋邏輯。

**複雜度**：🟡 中  
**影響面**：🟡 中  

---

## D. 效能與穩定性

### D1. Selector 自動修復（Self-Healing Selectors）

**描述**：當 Selector Health Check 偵測到 CSS 選擇器失效時，自動嘗試一組備用選擇器，而不是靜靜地過濾失效。

**技術可行性分析**：
- 🟡 可以在 `selectors.ts` 中為每個重要選擇器提供「備用選擇器陣列」，並在 `validateSelectors()` 失敗時自動切換。
- 難點在於如何判斷「選擇器有命中但命中錯誤的元素」。

**複雜度**：🟡 中  
**影響面**：🟢 大（大幅提升跨版本穩定性）  

---

### D2. MutationObserver 防洪機制改善

**描述**：目前當 mutations 超過 100 個時會切換到全頁掃描模式（Full Scan）。可以改為智慧分級：50~100 用增量，>100 用 requestIdleCallback 的全掃。

**技術可行性分析**：
- 🟢 修改 `video-filter.ts` 的 `processMutations()` 邏輯即可。
- 效能敏感區域，需要仔細測試。

**複雜度**：🟢 低  
**影響面**：🟡 中  

---

### D3. 記憶體洩漏監控

**描述**：長時間使用後，`SUBSCRIBED_CHANNELS` 與 `FilterStats` 可能無限增長。應加入上限控制。

**技術可行性分析**：
- 🟢 `SUBSCRIBED_CHANNELS`：若訂閱數超過 N（如 500）則停止新增。
- `FilterStats`：已有 `reset()`，可考慮定時重置（如每小時）或以 LRU 限制詳細記錄數量。

**複雜度**：🟢 低  
**影響面**：🟡 中  

---

### D4. 單元測試覆蓋率提升

**描述**：目前測試覆蓋主要集中在 `filter-engine`、`config-manager`、`filter-test`。`interaction.ts`、`subscription-manager.ts`、`style-manager.ts` 的測試相對薄弱。

**技術可行性分析**：
- 🟢 使用 `jsdom` 環境即可補充大部分測試。
- `subscription-manager.ts` 的掃描邏輯需要 mock DOM 結構。

**複雜度**：🟡 中  
**影響面**：🟢 大（提升重構信心）

## E. 統計與可觀測性

### E1. 按頻道統計過濾次數

**描述**：在統計面板顯示「哪些頻道最常觸發過濾」，讓使用者知道哪些頻道該加入黑名單、哪些該加入白名單。

**技術可行性分析**：
- 🟢 在 `FilterStats.record()` 傳入 channel name 並記錄即可。
- 需要修改 `StatsData` 介面加入 `byChannel: Record<string, number>`。
- `LazyVideoData` 的 channel 需要傳入 `hideElement()`。

**複雜度**：🟡 中  
**影響面**：🟡 中  

---

### E2. 過濾趨勢圖（簡易文字折線圖）

**描述**：記錄每個 session 的過濾次數，在統計面板以 ASCII 折線圖顯示過去幾次的趨勢。

**技術可行性分析**：
- 🟡 需在 GM Storage 中持久化每次 session 的統計摘要。
- ASCII 折線圖可以用純字串拼接實作。
- 複雜度不高，但對使用者的實用性有限。

**複雜度**：🟡 中  
**影響面**：🟢 低  

---

### E3. 健康報告（Health Report）

**描述**：在系統選單新增「健康報告」，一次顯示：選擇器狀態、最後掃描時間、訂閱數量、過濾統計、OpenCC 載入狀態等。

**技術可行性分析**：
- 🟢 純資訊彙整，不需要新功能，只需在 `UIManager` 新增一個 `showHealthReport()` 方法。

**複雜度**：🟢 低  
**影響面**：🟡 中（對開發者/進階使用者很有用）

## F. 整合與同步

### F1. 雲端同步設定（透過 GitHub Gist）

**描述**：讓使用者授權 GitHub Gist，自動上傳/下載設定，實現跨裝置同步。

**技術可行性分析**：
- 🔴 需要 OAuth 或 Personal Access Token（PAT）。
- Tampermonkey 的 `GM_xmlhttpRequest` 可以打 GitHub API，但 PAT 儲存在 GM Storage 有安全疑慮。
- 整個功能複雜度非常高，且安全性需要仔細設計。

**複雜度**：🔴 高  
**影響面**：🟡 中  

---

### F2. 與 uBlock Origin 規則共享

**描述**：讓使用者可以將「頻道黑名單」導出為 uBlock Origin 的自訂規則格式（Cosmetic Filter），互補使用。

**技術可行性分析**：
- 🟢 uBlock 的 Cosmetic Filter 格式是純文字（`youtube.com##ytd-rich-item-renderer:has-text(頻道名稱)`），可以從 CHANNEL_BLACKLIST 轉換。
- 只需一個匯出函式，複雜度低。

**複雜度**：🟢 低  
**影響面**：🟡 中  

---

### F3. SponsorBlock 整合

**描述**：整合 SponsorBlock API，讓腳本可以過濾掉被大量標記為「無聊/填充內容」的影片。

**技術可行性分析**：
- 🔴 需要對 SponsorBlock API 發送請求（`GM_xmlhttpRequest`），且要做大量快取避免頻繁請求。
- SponsorBlock 的標記是以影片 ID 為單位，首頁不顯示影片 ID，需要從 URL 提取。
- 法律/使用條款問題需要確認。

**複雜度**：🔴 高  
**影響面**：🟢 大
